## medium: Admin dashboard defaults to production API when configuration is missing

id: fnd_sig-feat-library-53dc6de542-a58c_8e6e6511d3
category: build-release
confidence: high
triage: risk
status: open
feature: Node source frontend/app/admin (feat_library_53dc6de542)

evidence:
- frontend/app/admin/ceo/page.tsx:6 (API_URL)
- frontend/app/admin/ceo/page.tsx:66-70 (fetchDashboard)
- frontend/app/admin/ceo/page.tsx:318-322 (triggerSalesAgent)

If NEXT_PUBLIC_API_URL is omitted in a preview, local, or staging build, this client-side admin page silently sends dashboard reads and sales-agent mutations to the production Railway API. Because triggerSalesAgent performs a POST, a misconfigured non-production frontend can execute production admin actions rather than failing closed.

recommendation:
Remove the production fallback for admin API calls. Require NEXT_PUBLIC_API_URL at build/runtime or derive the API base from a same-environment relative route, and fail visibly when it is not configured.

test analysis:
No tests are linked for this feature, and there is no test asserting API base selection under missing environment configuration.

suggested regression test:
Add a test or build-time config validation that fails when NEXT_PUBLIC_API_URL is absent for the CEO admin page, and verifies sales-agent actions target the configured base URL.

minimum fix scope:
Change API_URL configuration in frontend/app/admin/ceo/page.tsx and add configuration validation for the frontend admin build.

## medium: Admin pages render without route-level authentication

id: fnd_sig-feat-library-53dc6de542-18cb_53f0c0fbc1
category: security
confidence: medium
triage: risk
status: open
feature: Node source frontend/app/admin (feat_library_53dc6de542)

evidence:
- frontend/app/admin/layout.tsx:1-7 (AdminLayout)
- frontend/app/admin/buffer/page.tsx:20-24 (BufferAdminPage)
- frontend/app/admin/ceo/page.tsx:57-58 (CEODashboard)

The shared /admin layout does not enforce sign-in or admin authorization before rendering child pages. The CEO page only discovers authorization after rendering client code and calling the backend, while the Buffer admin page has no Clerk token or explicit authorization check at all before exposing scheduling and deletion controls. Backend APIs may still reject unauthorized mutations, but the admin UI itself is not protected at the route boundary, which is a concrete permission gap for an admin source group.

recommendation:
Add a route-level guard in frontend/app/admin/layout.tsx or middleware for /admin that requires an authenticated Clerk session and verifies the user is authorized for admin access before rendering any admin child route. Keep backend authorization checks as defense in depth.

test analysis:
No tests are linked for this feature, and simple-test.tsx only renders static route content without exercising authenticated or unauthenticated admin access.

suggested regression test:
Add an admin route access test that renders or navigates to /admin/buffer and /admin/ceo with no session and verifies the user is redirected or denied before admin controls are rendered.

minimum fix scope:
Protect the /admin layout or a /admin middleware matcher, and cover both owned admin pages.

## medium: Loading state starts after token retrieval, allowing duplicate onboarding submissions

id: fnd_sig-feat-library-6e2f59f09b-907c_6ae2cddc6d
category: bug
confidence: medium
triage: risk
status: open
feature: Node source frontend/app/onboarding (feat_library_6e2f59f09b)

evidence:
- frontend/app/onboarding/page.tsx:43-44 (handleNext)
- frontend/app/onboarding/page.tsx:51-52 (handleNext)
- frontend/app/onboarding/page.tsx:253-254

The button remains enabled while getToken() is pending because loading is not set until after token retrieval. Multiple rapid clicks can enter handleNext before loading flips true, causing duplicate POSTs to /api/onboarding/step/1 or /api/onboarding/step/2. That can create duplicate onboarding side effects or conflicting saved state depending on backend behavior.

recommendation:
Set loading true before awaiting getToken for submitting steps, or guard at the start of handleNext with if (loading) return. Limit the guard to steps that submit network requests if desired.

test analysis:
No linked tests exercise rapid repeated clicks or assert that each onboarding step submits at most once.

suggested regression test:
Add a component test that mocks getToken as a delayed promise, rapidly clicks Next twice on step 1, resolves the promise, and asserts fetch was called once.

minimum fix scope:
Update handleNext in frontend/app/onboarding/page.tsx to set or check loading before any awaited work for steps 1 and 2.

repro:
On step 1 or 2, click Next repeatedly before Clerk token resolution completes. Multiple handler invocations can pass the disabled guard and send repeated POST requests for the same step.

## medium: Terms acceptance is enforced only in client state before starting checkout

id: fnd_sig-feat-library-5c82b99f93-d3cc_060256267c
category: api-contract
confidence: medium
triage: contract-mismatch
status: open
feature: Node source frontend/app/geo (feat_library_5c82b99f93)

evidence:
- frontend/app/geo/page.tsx:14-30 (handleGeoCheckout)
- frontend/app/geo/page.tsx:143-160 (GeoPage)

The checkout flow blocks the button and handler until local React state says terms were accepted, but the API call only sends { plan: 'geo' }. A user or script can call /api/billing/addon-checkout directly without accepting terms, and the backend receives no acceptance flag, version, or timestamp from this page to enforce or record the contract. If terms acceptance is required before paid checkout, client-only gating is bypassable and creates an API contract/compliance gap.

recommendation:
Move terms enforcement/recording to the server-side checkout endpoint, or include an explicit acceptance payload that the backend validates and persists before creating the Stripe checkout session.

test analysis:
No linked tests were included for this feature, and the current page-level behavior only tests local UI state if tested at all; it would not prove the endpoint rejects direct calls without acceptance.

suggested regression test:
Add an endpoint-level test that POST /api/billing/addon-checkout for plan geo fails unless the request/user has a recorded accepted terms version, and succeeds only after acceptance is recorded.

minimum fix scope:
Backend checkout contract plus this page's checkout request shape if the server requires explicit acceptance metadata.

repro:
Call POST /api/billing/addon-checkout with an authenticated token and body {"plan":"geo"} without interacting with the checkbox; this page provides no server-verifiable acceptance data in the request.

## low: PostHog is initialized unconditionally with a hard-coded production project key

id: fnd_sig-feat-library-27665937d9-d422_346e977e37
category: build-release
confidence: medium
triage: risk
status: open
feature: Node source frontend/app/:provider (feat_library_27665937d9)

evidence:
- frontend/app/providers.tsx:8-15 (PHProvider)
- frontend/package.json:6-9 (scripts)

The provider always initializes PostHog when mounted and uses a literal project key/api host instead of environment-specific configuration or an enablement guard. In a Next frontend this means local development, preview, staging, and production builds all report to the same PostHog project whenever this provider is rendered. That can pollute production analytics and may capture unexpected non-production user/session metadata despite session recording and autocapture being disabled.

recommendation:
Move the PostHog key and host to public environment variables and initialize only when analytics is explicitly enabled for the current deployment environment. Consider skipping initialization when the key is absent.

test analysis:
No linked tests are provided for this feature, and there is no assertion around environment-specific analytics configuration or disabled development behavior.

suggested regression test:
Add a provider test that mocks posthog.init and verifies it is not called when the public key or analytics enablement flag is absent, and is called with values from environment configuration when enabled.

minimum fix scope:
Update frontend/app/providers.tsx to read PostHog configuration from environment variables and guard initialization; add a focused test for enabled/disabled initialization behavior.
